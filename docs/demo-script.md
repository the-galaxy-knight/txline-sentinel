# Judge Demo Script

This script is designed for a 3-5 minute hackathon demo video. The goal is to prove three things quickly:

- TxLINE Sentinel can connect to TxLINE live/devnet inputs.
- Replay scenarios exercise the same backend pipeline as live ingestion.
- The agent produces deterministic, auditable signals with follow-through tracking.

## Demo Positioning

Suggested opening line:

```text
TxLINE Sentinel is an autonomous odds intelligence agent. It ingests TxLINE odds and score feeds, detects meaningful market movement, explains why the signal matters, and tracks whether each signal was predictive after 5, 10, and 15 minutes.
```

Be explicit about live versus replay:

```text
Live TxLINE ingestion is configured and visible in the dashboard. For a deterministic judge demo, I use replay scenarios that run through the same normalizer, market state, detector, scorer, explanation, persistence, evaluation, and SSE pipeline.
```

## Pre-Demo Setup

Backend `.env` should contain TxLINE credentials and live mode:

```env
TXLINE_BASE_URL=https://txline-dev.txodds.com
TXLINE_GUEST_JWT=<set>
TXLINE_API_TOKEN=<set>
TXLINE_ODDS_STREAM_PATH=/api/odds/stream
TXLINE_SCORES_STREAM_PATH=/api/scores/stream
INGESTION_MODE=live
```

Start the backend:

```powershell
cd backend
.venv\Scripts\python -m uvicorn app.main:app --reload --port 8000
```

Start the frontend:

```powershell
cd frontend
npm run dev
```

Open:

```text
http://localhost:3000
```

## Historical Replay Scenarios

Yes, TxLINE provides historical replay/data endpoints. The World Cup free tier documentation advertises full access to historical data for past match and event analysis, and the API reference exposes historical 5-minute interval endpoints for odds and scores.

Current local replay files in this repo are **TxLINE-style demo replays**, not automatically downloaded TxLINE historical intervals. Once we fetch historical odds and score intervals from TxLINE and save them as replay JSON, those scenarios should be labeled as **TxLINE historical replays**.

Use **historical-inspired synthetic replays** only when the match pattern is inspired by a previous match but the odds timeline is manually authored.

Good historical-inspired examples:

- A late equalizer causing a rapid reversal in winner probability.
- A red card triggering bookmaker divergence before consensus catches up.
- A penalty or disallowed goal causing a short-lived overreaction.
- A long no-score period where one side shortens sharply despite no visible score event.
- A post-goal market reaction where probability moves in multiple waves.

When adding synthetic historical-inspired scenarios:

- Use real match names only for broad narrative context.
- Do not claim exact historical odds unless sourced.
- Keep `FixtureId` synthetic, for example `demo-arg-ned-2022-inspired`.
- Use TxLINE-style payload fields so the normalizer path remains representative.
- Include both score events and odds events if the signal depends on match context.
- Make sure at least one odds event crosses a documented threshold.

When adding TxLINE-derived historical scenarios:

- Pull odds updates from TxLINE historical interval endpoints.
- Pull score updates from TxLINE historical interval or full fixture sequence endpoints.
- Preserve TxLINE fixture IDs, message IDs, timestamps, bookmakers, prices, and percentages.
- Add metadata in the replay description that states the source is TxLINE historical data.
- Avoid editing the raw payload except for harmless fixture naming or demo metadata.

The backend includes a CLI generator:

```powershell
cd backend
.venv\Scripts\python -m app.cli build-historical-replay `
  --start 2026-07-09T12:00:00Z `
  --end 2026-07-09T12:30:00Z `
  --fixture-id 18143850 `
  --scenario-name txline_fixture_18143850_history `
  --display-name "TxLINE historical fixture 18143850"
```

The same workflow is available without CLI from the frontend:

```text
/replay -> Build TxLINE Historical Replay
```

The frontend starts a background build job and polls progress. During recording, show:

- the estimated interval/request count before starting,
- the queued or running job status,
- intervals completed versus requested,
- odds and score events discovered so far,
- the current 5-minute historical interval,
- the saved replay path after completion.

For long ranges, explain the scale before starting. A range from `2026-06-01T12:00:00Z` to `2026-07-10T12:00:00Z` is 39 days, which is 11,232 five-minute intervals and 22,464 TxLINE requests because the builder fetches odds and scores separately for each interval.

This fetches:

```text
GET /api/odds/updates/{epochDay}/{hourOfDay}/{interval}
GET /api/scores/updates/{epochDay}/{hourOfDay}/{interval}
```

Official references:

- TxLINE World Cup free tier: https://txline.txodds.com/documentation/worldcup
- Historical odds interval: https://txline.txodds.com/api-reference/odds/get-a-json-array-of-all-odd-updates-from-a-specific-historical-5-minute-interval
- Historical scores interval: https://txline.txodds.com/api-reference/scores/get-a-json-array-of-all-score-updates-from-a-specific-historical-5-minute-interval-no-live-data-is-returned

and writes a replay scenario with:

```json
{
  "source": "txline_historical"
}
```

Replay files live here:

```text
backend/app/data/replay_scenarios
```

A new replay file only needs:

```json
{
  "name": "Example historical-inspired scenario",
  "description": "Synthetic odds path inspired by a previous World Cup match pattern.",
  "fixture_id": "demo-example-001",
  "events": [
    {
      "offset_ms": 0,
      "event_type": "score",
      "payload": {
        "FixtureId": "demo-example-001",
        "Ts": "2026-07-09T12:00:00Z",
        "Action": "match_started",
        "Participant1Score": 0,
        "Participant2Score": 0,
        "GameState": "1H"
      }
    }
  ]
}
```

After adding a JSON file, restart the backend or refresh replay scenarios:

```powershell
Invoke-RestMethod http://localhost:8000/api/replay/scenarios
```

## Demo Risk Notes

- Devnet may not have live match activity at review time.
- Replay is therefore the deterministic product demo path.
- The dashboard must still show live TxLINE configuration and stream health.
- SQLite and in-memory SSE/state are MVP choices; they are documented production-readiness limitations.
- The product does not execute trades. It generates intelligence signals and tracks predictiveness.
